How the Console Runs

Frontend to Backend
The frontend collects selected netlists, selected algorithms and DSE metrics, then posts JSON to the matching Flask endpoint. The backend parses the netlist again for each algorithm so every engine starts from the same circuit state.
UI selection -> /api/run or /api/dse* -> parse_netlist() -> levelize() -> engine.run() -> format_result()
Basic run uses the event-driven simulator and then formats node, PI, PO and fault counts.
DSE runs compare algorithms or kernels and return per-algorithm metrics plus overlap summaries.
Iterative mode repeats the same comparison N times and aggregates min, max, average and standard deviation.

Algorithm Briefs

Core Engines
Basic Flow /api/run simulate_event_driven()

Purpose

Builds a parsed circuit, assigns default inputs, runs event-driven simulation and reports the raw circuit state. It is the simplest end-to-end path and is also used to generate or reuse a visual netlist image.

Key functions: parse_netlist(), levelize(), assign_default_inputs(), simulate_event_driven(), format_basic_result().

Read the output as: circuit size, generated faults, PI/PO values and available image sources.

D /api/run or /api/dse* DAlgorithmEngine

Purpose

Runs the classic D-algorithm ATPG engine on the levelized circuit. In DSE #1 it is compared directly with PODEM and in DSE #3 it is compared with D_EXHAUSTIVE.

Key functions: DAlgorithmEngine(circuit), run_engine_with_memory(), dse_algo_metrics().

Read the output as: coverage, backtracks, time, memory, final vector count and the detected-versus-undetected fault split.

PODEM /api/run or /api/dse* PODEMEngine

Purpose

Runs PODEM, optionally with heuristics. The heuristic-enabled and heuristic-free versions are compared in DSE #2.

Key functions: PODEMEngine(circuit, use_heuristics=True/False), run_engine_with_memory(), detected_fault_set().

Read the output as: performance tradeoffs between heuristic search and baseline search, plus overlap in detected faults.

DSE #1 to #3

ATPG Comparison
DSE #1: D vs PODEM

Shows whether PODEM improves coverage, time or backtracking compared with the baseline D engine. The iterative endpoint averages those same metrics across repeated runs.

DSE #2: PODEM vs PODEM_NO_HEUR

Isolates the effect of heuristics inside the same search strategy. This is the cleanest comparison for judging search guidance.

DSE #3: D vs D_EXHAUSTIVE

Compares the baseline D engine against the more exhaustive variant so you can see whether extra search effort buys additional detection.

DSE #4 to #5

Simulation and Fill Policy
DSE #4: SIMULATE vs EVENT_DRIVEN

Uses the Basic Flow circuit and compares the two simulations. Time and memory matter most here because the fault-detection result is the same circuit semantics.

DSE #5: Fill-bit policies

Starts from the detected vectors produced by D and then fills X bits using 0-fill, 1-fill or deterministic random-fill. The interesting metrics are test vector count, toggle count, peak switching and fill overhead.

Iterative mode is supported for all DSE routes. The backend repeats the same comparison N times and returns aggregate statistics instead of a single execution snapshot.

Reading the Metrics

What Matters
Coverage (%) Percentage of faults detected by the algorithm on the current netlist.
Time (ms) Wall-clock runtime for the engine or simulation execution.
Backtracks Search backtracking effort, especially important for D and PODEM comparisons.
Memory (KB) Peak traced memory during execution.
Detected faults Faults where detected=true in the per-fault result list.
Undetected faults Faults where detected=false, useful for seeing what still escapes the ATPG search.

What Happens When You Click Run

Execution Flow
1. The browser gathers selected netlists, algorithms and metrics from the left panel.
2. The frontend posts JSON to the matching route such as /api/run or /api/dse-d-variants-iterative.
3. The backend parses the netlist again so each algorithm starts from the same baseline circuit.
4. The relevant ATPG engine or simulation kernel runs and captures runtime, memory, vectors and faults.
5. The result formatter packages the response into the card-friendly shape shown in the console.
run button -> fetch() -> Flask route -> parse_netlist() -> engine.run() -> format_result()/dse_algo_metrics()
